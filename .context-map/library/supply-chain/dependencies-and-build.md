# Constraints — supply-chain (dependencies + build)

1. Lockfile (`package-lock.json`, `poetry.lock`, `Cargo.lock`) MUST be committed and updated together with the manifest in the same commit; a lockfile that lags the manifest produces non-reproducible installs across machines and CI.
2. CI install steps MUST NOT use `npm install` (which mutates the lockfile); use `npm ci` (or `pnpm install --frozen-lockfile`, `poetry install --no-update`) so a stale lockfile fails loud instead of silently drifting.
3. Setup scripts MUST NOT use `curl ... | bash` or `wget ... | sh` patterns to install tooling; the remote payload can change between commit and execution and there is no signature check. Pin the binary in a package manager or vendor it.
4. CI MUST NOT install global packages (`npm i -g`, `pip install --user`); globals are invisible to the lockfile, so the runner's snapshot drifts every week and breaks reproducibility.
5. Build and CI workflows MUST source secrets from the platform secret store (GitHub Actions secrets, Vercel env vars, Convex env), never from a checked-in `.env` or a `.envrc`; even a deleted-from-history secret leak is permanent.
6. Dockerfiles MUST pin the base image by digest (`FROM node@sha256:...`) for production builds; `FROM node:20` re-pulls a different binary every time the upstream tag moves and turns 'works yesterday' into 'broken today' with no diff to point at.
7. Dockerfiles MUST NOT run as root in the final stage; use `USER nonroot` (or the distroless equivalent) so a process compromise doesn't immediately yield container-level capabilities.
