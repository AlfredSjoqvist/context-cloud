# Constraints — validation (schema at trust boundaries)

1. Every untrusted input crossing into the service (HTTP body, query string, MCP tool input, webhook payload, queue message) MUST be parsed through a schema validator (Zod, Pydantic, JSON Schema) before any business logic reads a field; reading `input.foo` before validation is the foothold for type-confusion bugs.
2. Handler signatures MUST NOT accept `Record<string, unknown>` or `dict[str, Any]` as the public type; that shape says `I have not validated this` to every reader and lets unchecked fields propagate through three function calls before crashing.
3. Validation failures MUST return HTTP 422 with a structured error payload listing the failing field path; a generic 400 forces the caller to guess which field they got wrong, and a 500 leaks an exception message into the client.
4. Schemas MUST be strict about extra fields (`zod.strict()`, Pydantic `extra='forbid'`); accepting unknown fields silently lets a typo in the caller's payload sit there for months without error.
5. JSON parsing of an untrusted body MUST happen inside the schema validator (`safeParse(body)`), not before — a `JSON.parse` followed by a try/catch handles malformed JSON but loses the schema-validation step's structured error path.
6. String / array fields with no max-length bound MUST set one (typically 1–10 KB for strings, 100 elements for arrays); unbounded fields are the standard memory-exhaustion vector for any service that accepts JSON.
