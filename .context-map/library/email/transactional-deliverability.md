# Constraints — email (transactional + deliverability)

1. Outbound email MUST be sent from a domain whose DNS publishes valid SPF, DKIM, and DMARC records; missing any one drops your inbox-placement rate by 30%+ at Gmail / Outlook and turns transactional mail into a slow-burning incident.
2. Password-reset emails MUST NOT include the new password (or the old one, or a recoverable encoding); send a single-use token URL instead. Plaintext credentials in inboxes are the most-replayed credential leak in any breach corpus.
3. Password-reset and email-verification tokens MUST be single-use, expire within an hour, and be revoked on use; reusable or long-TTL tokens become permanent backdoors the moment an inbox is compromised.
4. Per-recipient sending MUST be rate-capped (typically 1/min, 10/hour); without a cap, a bug or a malicious caller can mail-bomb a single user and get your sending domain blacklisted across the recipient's organisation.
5. The transactional-mail send path MUST honor SES/SendGrid/Postmark bounce + complaint webhooks; ignoring them keeps re-sending to invalid addresses and accelerates the path to a spam-classifier blackhole.
6. Transactional emails (password reset, MFA codes, account changes) MUST NOT include open-tracking pixels or click-tracking redirects; those rewrite the URL and trigger anti-phishing filters that quarantine the email in corporate Outlook.
