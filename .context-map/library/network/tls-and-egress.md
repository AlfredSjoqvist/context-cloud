# Constraints — network (TLS and egress)

1. TLS verification MUST NOT be disabled in production (`rejectUnauthorized: false`, `verify=False`, `--insecure`); a single skipped verification anywhere in the request graph turns every downstream call into an opportunity for an on-path attacker.
2. Outbound HTTPS clients MUST pin the minimum TLS version to 1.2 or higher; defaults still negotiate TLS 1.0/1.1 in some Node and Python versions, and 1.0/1.1 are deprecated and exploitable.
3. Production services MUST handle DNS resolution failure as a transient retryable error (with backoff), not a permanent crash; intermittent DNS failures are the most common cloud-platform fault mode and are usually resolved by the next cache refresh.
4. Outbound requests whose URL comes (in whole or part) from caller input MUST NOT be issued without an allow-list check on the resolved host; SSRF against internal metadata endpoints (169.254.169.254 on AWS / GCP) is the standard cloud-credential exfiltration vector.
5. Outbound HTTP requests MUST set BOTH a connect timeout AND a read timeout (typically 5–10 s connect, 15–60 s read); a single missing timeout on a slow upstream stalls the calling request indefinitely and exhausts the connection pool.
6. Outbound HTTP requests with credentials (Authorization header, cookies) MUST NOT auto-follow redirects across origin boundaries; the standard fetch / requests behaviour leaks credentials to whatever host the redirect target points at.
