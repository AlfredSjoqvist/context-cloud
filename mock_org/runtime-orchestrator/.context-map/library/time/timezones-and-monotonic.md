# Constraints — time (timezones + monotonic clocks)

1. Datetimes persisted to storage MUST be UTC and serialised in ISO 8601 with the explicit `Z` suffix or `+00:00` offset; a naive `datetime.now()` (no tzinfo) leaks the server's local clock into the database and silently shifts an hour every March and November.
2. Time-of-day comparisons (e.g. `hour < 9`) MUST be done in the user's timezone, never the server's; a `Europe/London` user opening an `America/Los_Angeles` server sees their 8 AM marked as evening, and feature flags gated on time-of-day silently misfire.
3. Elapsed-time measurements MUST NOT use wall-clock arithmetic (`Date.now()` end - `Date.now()` start, `time.time()` deltas); use `performance.now()` / `time.monotonic()` so an NTP step or DST transition mid-measurement doesn't yield a negative duration.
4. Date parsing MUST require an explicit format (`strict=True`, `parseISO`, `Temporal.PlainDate.from`); the lenient defaults in `Date(...)`, `dateutil`, and `moment(...)` accept plausible-but-wrong strings and silently produce garbage dates a year off.
5. Adding `N days` to a date MUST use a calendar-aware operation (`add({days: N})`, `relativedelta(days=N)`); adding `N * 86400` seconds to a Unix timestamp gives the wrong answer across DST boundaries by exactly one hour.
6. Database column types MUST be `TIMESTAMPTZ` (Postgres) or `TIMESTAMP WITH TIME ZONE` equivalents; the unqualified `TIMESTAMP` type stores values without timezone info and turns every read into a guess about which clock wrote them.
