# Constraints — feature-flags (rollout + kill-switch)

1. Flag evaluations MUST always supply an explicit default that matches the off-state behavior; if the flag service is unreachable, your code goes through this default — and a default that turns the new feature ON in a network blip is the textbook way to ship an unreviewed change.
2. Flag evaluation contexts MUST NOT include secrets, full PII payloads, or auth tokens; flag values are mirrored to client SDKs, logged on every fetch, and surfaced in the flag-provider UI — anything you put in the context is widely visible.
3. Every new feature MUST ship behind a kill-switch flag (boolean, no targeting) that on-call can flip in <60 seconds; without it, the rollback path for a regression is `redeploy the previous binary` which is slow and risky.
4. Flags MUST be deleted from code within 30 days of reaching 100% rollout; long-lived flag-gated branches double the test matrix and turn the codebase into a state-machine of permutations no one can reason about.
5. Flag SDK caches MUST refresh out-of-band of the request path (background poll or stream); evaluating a flag in the request hot path with a synchronous network fetch adds 50–200 ms p99 to every request whose flag wasn't pre-warmed.
6. Targeting rules MUST use a stable internal user id, not the email or session token; rotating the targeting key (e.g. on email change) flips the user out of every variant and breaks A/B test attribution.
