# Constraints — db (transactions and migrations)

1. Database queries that include any caller-controlled value MUST use parameter binding (`$1`/`?`/`:name`); string-concatenated SQL is the canonical SQL injection sink even when the value `looks safe` to the author.
2. Multi-row writes that must succeed or fail together MUST run inside a single transaction; emitting writes individually leaves the database in a partial state on the first failure with no rollback path.
3. Long-running operations (LLM calls, HTTP requests, file reads larger than 1 MB) MUST NOT execute while a database transaction is open; a held transaction blocks vacuum, escalates locks, and stalls every other writer on the same row.
4. Endpoints that mutate user-visible state MUST accept and persist an idempotency key (header or body); without one, a client retry after a partial-network failure double-charges, double-creates, or double-sends.
5. Schema changes that drop or rename a column MUST ship as two deploys: first a backward-compatible deploy that writes both old and new shapes, then a follow-up deploy that removes the old shape; single-deploy drops break every replica still serving the previous binary.
6. Adding a `NOT NULL` column to an existing table MUST include a default expression OR run as a three-step migration (add nullable → backfill → set NOT NULL); a bare `ADD COLUMN ... NOT NULL` rewrites every row under an exclusive lock and times out on tables larger than a few million rows.
