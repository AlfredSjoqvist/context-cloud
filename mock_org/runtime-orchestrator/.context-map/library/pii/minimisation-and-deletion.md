# Constraints — pii (minimisation + deletion)

1. User-identifier fields (email, phone, full name, address) MUST only be collected when there is a documented user-facing purpose for that field; collecting `just in case` is a GDPR Article 5(1)(c) violation and a breach-amplifying surface area.
2. PII columns (email, phone, address, government id) MUST be encrypted at rest with a key managed outside the database (KMS / HSM); database-only encryption protects against a stolen disk but not against a stolen DB credential.
3. User-deletion endpoints MUST cascade to every table holding the user's PII (analytics events, audit logs, support tickets, mailing-list rows) within a documented SLO; partial deletion across systems is the most common GDPR Article 17 violation that triggers a regulator notice.
4. Reads of another user's PII (admin tools, support tooling) MUST land an audit-log row with operator id, target user id, justification, and timestamp; without an audit log, an internal-actor data exfiltration is invisible until a customer notices.
5. User-facing identifiers in URLs and emails MUST NOT be sequential or guessable (`/user/12345/profile`); use opaque random ids or UUIDs so an attacker can't enumerate user accounts by incrementing the integer.
6. User-data-export endpoints MUST return data in a structured format the user can re-ingest (JSON, CSV, ZIP) and MUST be rate-limited per account; without rate-limiting, an attacker who phishes a session can dump the account at line speed.
