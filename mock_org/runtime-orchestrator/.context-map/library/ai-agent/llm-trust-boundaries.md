# Constraints — ai-agent (LLM trust boundaries)

1. Untrusted text concatenated into an LLM prompt MUST be wrapped in an explicit delimiter (e.g. `<user_input>...</user_input>`) AND the system prompt MUST instruct the model to treat anything inside that delimiter as data, not instructions; without both, prompt injection from a tool output or user message is the canonical exfiltration vector.
2. Tool outputs that flow back into the LLM MUST be schema-validated (length cap, allowed types, no raw HTML/markdown unless explicitly intended); a tool that returns attacker-controlled text is a one-step prompt injection if that text is fed back without sanitisation.
3. Per-request token budgets MUST be enforced as hard ceilings (max input tokens, max output tokens, max tool-call iterations); without them an agent that hits a thinking loop runs the bill into the thousands before anyone notices.
4. Side-effecting agent tools (writes, deletes, sends, payments) MUST require a confirmation argument or land in a queued-for-review state when invoked from an autonomous cycle; an LLM that decides to retry sends the same delete-account call three times by default.
5. When the downstream consumer is code (not a human), prefer structured-output mode (JSON Schema, function calling, Pydantic) over free-text parsing; regex-extracting fields from prose is the standard cause of `it worked yesterday` agent regressions when the model wording shifts.
6. Every LLM call MUST be logged with prompt id, model name, input + output token counts, latency, and a hash of the input; without this trace, debugging a specific agent failure means re-running the whole chain at non-zero cost and possibly different outputs.
7. LLM prompts MUST NOT include the raw bearer token, signed cookie, or API key of the calling user; the prompt is logged at the LLM provider, may appear in transcripts, and is preserved for fine-tuning analytics — leaking to surfaces you don't control.
