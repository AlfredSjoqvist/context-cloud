# Constraints — caching (TTL + invalidation)

1. Every cache entry MUST set an explicit TTL; an unbounded cache grows until the process OOMs and a stale entry served forever is the same shape as a wrong answer that nobody can find the source of.
2. Cached responses MUST NOT mix authenticated and anonymous content under the same key; the next anonymous reader gets the previous authenticated reader's data, and your cache becomes a credential-leaking sink.
3. Writes that change cached content MUST invalidate (or version-bump) the corresponding cache key in the same code path that performs the write; relying on TTL alone makes every reader during the TTL window see stale data, which is the most common form of `it works for me but the user sees the old value`.
4. Cache misses for hot keys MUST coalesce concurrent regenerations (single-flight, lock-while-recompute, or probabilistic early refresh); without this, the moment a hot key expires every replica hits the origin in parallel and the origin falls over.
5. Cached values MUST be serialised with a stable cross-language schema (versioned JSON, protobuf, msgpack); a language-native binary serializer couples the cache wire format to the language version and breaks the moment a worker upgrades.
6. Cache backend failures (Redis down, connection refused) MUST fall back to the origin instead of returning an error to the user; a cache is a latency optimisation, not a hard dependency, and treating it as one converts every cache hiccup into a user-visible outage.
