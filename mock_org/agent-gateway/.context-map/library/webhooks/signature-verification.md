# Constraints — webhooks (signature verification + replay safety)

1. Webhook signature verification MUST check that the candidate signature buffer and the expected signature buffer have the same byte length BEFORE calling `crypto.timingSafeEqual`; the function throws on mismatched lengths and turns a 401-shaped failure into a 500-shaped one.
2. Signature comparison MUST use a constant-time primitive (`crypto.timingSafeEqual` in Node, `hmac.compare_digest` in Python); `===` and `==` leak prefix-equality timing and are exploitable across a network with enough samples.
3. Webhook handlers MUST reject events whose timestamp is older than a fixed tolerance (typically 5 minutes); without a timestamp window an attacker who captures one signed payload can replay it indefinitely.
4. Webhook bodies MUST be parsed inside a try/catch (or schema-validating equivalent like Zod's `safeParse`); a malformed payload that crashes the handler produces a 5xx that the upstream marks as failed and retries forever.
5. Webhook handlers MUST persist the upstream event id and short-circuit duplicate deliveries; every webhook provider re-sends events on a 5xx, so a non-idempotent handler double-applies state on every transient hiccup.
6. The webhook signing secret MUST come from the runtime environment (env var, secret manager), never from the request body or query string; secrets in URLs leak into proxy logs, browser history, and bug reports.
