# Constraints — state (durable + atomic)

1. File writes that represent durable state MUST be atomic (write to a sibling temp file, fsync, then rename); a process crash mid-`write_text`/`writeFileSync` leaves a half-written file that the next reader either deserialises into garbage or crashes on.
2. Atomic-write helpers MUST `os.fsync` the file descriptor before the `os.rename`; rename without fsync survives the rename but loses the file contents on a power loss because the data is still in the OS write-cache.
3. Process-local `set` / `dict` / `Map` MUST NOT be the source of truth for cross-request state (job-id dedup, idempotency keys, session counters); a single-replica deploy hides the bug, and the first time you scale out you double-process every event.
4. Serialised state MUST include a schema version field (`version: 1`); without one, every breaking change to the on-disk format requires a backfill migration that you cannot write because the writer doesn't know which version it just read.
5. Read paths that load a checkpoint or queue snapshot MUST cap the maximum file size (typically 10–100 MB depending on payload class); an unbounded read of an attacker-controlled file is a memory DoS.
6. Checkpoint stores MUST garbage-collect entries older than a documented retention window (typically 7–30 days); without GC, the directory grows until the inotify watcher / inode table fails — long after the last on-call rotation has forgotten the system exists.
