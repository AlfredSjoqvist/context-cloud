# Constraints — frontend-security (XSS, CSRF, CSP)

1. React components MUST NOT pass user-controlled strings to the raw-HTML escape hatch prop without first sanitising them through DOMPurify (or an equivalent allow-list HTML sanitiser); any path from user input to that prop is the canonical stored-XSS sink.
2. Code MUST NOT pass user-controlled strings to dynamic-evaluation APIs (`eval`, the `Function` constructor, `setTimeout`/`setInterval` with a string argument); each is a script-injection sink even when the input has been encoded for HTML, because they re-evaluate the string as JavaScript.
3. Production responses MUST set a Content-Security-Policy header with at minimum `default-src 'self'`, `script-src` without `'unsafe-inline'` / `'unsafe-eval'`, and `frame-ancestors 'none'` (or a tight allow-list); without CSP, a single XSS becomes an account takeover.
4. Cookie-authenticated mutating endpoints MUST require a CSRF token (double-submit cookie or anti-forgery token), OR be reachable only via `SameSite=Strict`/`Lax` cookies AND an `Origin`/`Referer` check; one of the three must be in place.
5. Session cookies MUST be set with `HttpOnly`, `Secure`, and `SameSite=Lax` (or `Strict`) flags; the absence of any one of these turns a session cookie into either a JS-readable secret, an HTTP-leakable secret, or a CSRF target.
6. Iframes that embed third-party or user-generated content MUST set `sandbox` (without `allow-same-origin` for untrusted sources) and an explicit `allow` list; an unsandboxed iframe runs in the parent's security context and can navigate the top window.
7. Server-side redirect destinations derived from query params or referer MUST be matched against an allow-list of internal paths; an open redirect is the standard phishing-credential-relay vector and degrades the trust of every link in your domain.
