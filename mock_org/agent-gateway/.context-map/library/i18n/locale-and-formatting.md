# Constraints — i18n (locale and formatting)

1. User-facing strings MUST NOT be assembled by concatenating translated fragments (`t('greeting') + ' ' + name + t('welcome')`); German, Japanese, and Arabic each reorder words in ways no concatenation pattern recovers. Use named placeholders inside one translation key (`t('greeting', {name})`).
2. Plural forms MUST go through a CLDR-aware plural rule (`Intl.PluralRules`, `react-i18next` plural key); `count === 1 ? 'item' : 'items'` covers exactly two of the six plural categories CLDR defines (zero, one, two, few, many, other) and silently mistranslates the rest.
3. Numbers and currency MUST format through `Intl.NumberFormat(locale, ...)` (or framework equivalent); a hard-coded `${amount.toFixed(2)}` ships `1,234.56` to users whose locale writes `1.234,56` and silently misreads decimals.
4. Dates MUST format through `Intl.DateTimeFormat(locale, ...)`; `2026-05-10` reads as Oct-5 in en-US contexts and May-10 in en-GB contexts when displayed without explicit formatting, and date inputs MUST `Intl.DateTimeFormat`-parse with the user locale, not the server locale.
5. Layouts MUST use logical CSS properties (`margin-inline-start`, `padding-inline-end`) instead of physical (`margin-left`, `padding-right`) so RTL locales (Arabic, Hebrew) mirror correctly; mixing physical properties with `dir="rtl"` half-mirrors the layout and produces unreadable nested rows.
6. Text inputs accepting names, addresses, and free-form fields MUST allow Unicode (no `^[a-zA-Z]+$` validators); ASCII-only validators reject every name with diacritics, every Cyrillic / CJK / Devanagari name, and every email containing IDN characters.
