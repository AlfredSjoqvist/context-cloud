# Constraints — sandbox (job resource budgets + isolation)

1. Job resource budgets MUST validate both an upper AND a lower bound for memory, CPU, and timeout; an unchecked `0` or negative value either crashes the runtime or starves the job into a hang that consumes a worker slot indefinitely.
2. Job timeout MUST be enforced inside the sandbox by an external process (signal, container OOM kill, or scheduler cancellation), not by the job's own code path; a job in an infinite loop never reaches its own timeout check.
3. Container images MUST be pinned by digest (`@sha256:...`) for production jobs; floating tags (`:latest`, `:v1`) silently change the binary across runs and make incident triage impossible.
4. Image references MUST be allow-listed against an explicit registry list (gcr.io/<org>/, ghcr.io/<org>/); accepting arbitrary registries is a supply-chain takeover vector for any caller who controls the image string.
5. Job commands MUST be passed as an arg list (`["python", "-m", "foo"]`), never as a shell string; a single shell-string sink is the canonical command-injection vulnerability for a job runner.
6. Sandbox cleanup MUST run inside a `finally` block (or equivalent), not at the end of the happy path; an exception during job execution would otherwise leak the sandbox container, file mounts, or temp directory.
