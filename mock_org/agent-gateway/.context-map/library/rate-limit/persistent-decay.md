# Constraints — rate-limit (persistent + decaying)

1. Rate-limit counters MUST decay over time; an unbounded monotonic counter saturates within hours of normal traffic and rejects every subsequent request as over-limit.
2. Rate-limit state MUST NOT live in a process-local `Map`, `Set`, or `Object`; multi-instance deployments split traffic across replicas, so a process-local counter measures one replica's slice — not the user's actual request rate.
3. Rate-limit storage MUST evict expired buckets; without eviction, the bucket map grows unbounded with the cardinality of the bucket key (per-user × per-route × per-IP) and OOMs the process before the limit ever fires.
4. Rate-limit responses MUST return HTTP 429 with a `Retry-After` header (seconds or HTTP-date); SDKs and browsers silently retry without this header and amplify the load that caused the rejection.
5. Rate-limit keys MUST include the authenticated principal when available; falling back to IP alone collapses every customer behind a corporate NAT into one bucket and rate-limits unrelated tenants.
