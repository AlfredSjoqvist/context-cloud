# Constraints — crypto (primitives + keys)

1. Code MUST NOT implement cryptographic primitives by hand (block ciphers, HMACs, KDFs, asymmetric signatures); use the language stdlib (`crypto`, `cryptography`) or libsodium. A homegrown primitive is a vulnerability surface that no eval will catch.
2. MD5 and SHA-1 MUST NOT be used for any security purpose (passwords, tokens, signatures, integrity); both have practical collisions. Use SHA-256 (or SHA-3) for integrity, BLAKE3 for fast hashing, and a KDF for password storage.
3. Stored passwords MUST be hashed with a memory-hard KDF (argon2id, scrypt, or bcrypt with cost ≥ 12); a fast hash like SHA-256 + salt is crackable at billions/sec on commodity GPUs.
4. Symmetric encryption MUST use an AEAD construction (AES-GCM, ChaCha20-Poly1305); CBC + HMAC-SHA-256 is correct only with constant-time MAC verification, and most hand-rolled implementations of that pair leak via padding-oracle.
5. Cryptographic randomness MUST come from `crypto.randomBytes` (Node), `secrets` (Python), `crypto/rand` (Go); `Math.random()`, `random.random()` are deterministic PRNGs trivially predictable from a few outputs.
6. Encryption keys MUST come from a managed secret store (KMS, HashiCorp Vault, cloud secret manager) and be rotated at a documented cadence; hard-coded keys, env-only keys without rotation, and keys committed to git are the most common single-line cause of a credential incident.
