# Constraints — payments (idempotency + money)

1. Charge / capture / refund calls MUST send a stable idempotency key (UUID derived from the user-action id, NOT a per-call random); a re-issued retry without the key double-charges the customer, and a per-call random key prevents the upstream from de-duping.
2. Monetary amounts MUST be stored and computed in the smallest currency unit as integers (cents, paise), never floats; `0.1 + 0.2 !== 0.3` is the textbook reason finance teams stop trusting your invoice totals.
3. Every monetary amount MUST be paired with an explicit currency code (ISO 4217); a bare `amount` field eventually gets cross-summed across currencies into a meaningless total that someone uses for forecasting.
4. Card data (PAN, CVV, expiry, magnetic stripe) MUST NOT touch any logger, structured-log field, transaction record, or analytics event; PCI-DSS makes that data ineligible to even traverse non-compliant systems, and a single log line that captures it scopes the entire host into PCI-DSS audit.
5. Webhook handlers from the payment provider (Stripe, Adyen) MUST verify the provider signature header AND check the event timestamp tolerance window AND dedupe on the event id; missing any of the three lets an attacker forge a paid-status flip on your account.
6. Refund-issuing endpoints MUST be authenticated AND audit-logged with the operator id; a refund endpoint reachable without authn is the canonical insider-fraud + abuse vector that reads as a bug for years before someone notices.
