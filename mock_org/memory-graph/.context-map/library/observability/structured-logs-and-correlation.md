# Constraints — observability (structured logs + correlation)

1. Production code paths MUST NOT call `console.log`, `console.warn`, or `console.error` directly; calls MUST go through a structured logger that emits one JSON object per line so log shippers (Loki, Datadog, Cloud Logging) can index fields.
2. Every log line that crosses a request boundary MUST include a correlation id (trace id, request id, or session id); without one, joining across a service crash plus its retries is impossible after the fact.
3. Log fields MUST use stable snake_case or camelCase names within a service; mixing `userId` and `user_id` for the same concept fragments dashboards and breaks alerts that filter on the field name.
4. Logs MUST NOT include the raw bearer token, signed cookie, API key, or PII fields (email, phone, full name) that a downstream tool pretty-prints into user-visible UIs; redaction MUST happen at log emission, not at log ingestion.
5. Counter and histogram names MUST follow `<domain>_<action>_<unit>` (e.g. `auth_login_attempts_total`, `db_query_duration_seconds`); stable naming is the contract dashboards depend on across service rewrites.
6. Errors MUST be logged with the actual error object (so `cause` chain is preserved), not just `error.message`; dropping the cause chain hides every stack frame below the catching layer.
