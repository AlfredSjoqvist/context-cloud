# Constraints — file-uploads (size, MIME, paths)

1. File-upload endpoints MUST enforce a max payload size at the body-parser level (e.g. `bodyParser({limit: '10mb'})`); a missing limit lets a single uploader OOM the process by streaming a multi-GB body.
2. MIME-type validation MUST NOT trust the `Content-Type` header from the client; the header is caller-controlled. Use a magic-byte sniff (`file-type` npm, `python-magic`) on the actual bytes.
3. Stored filenames MUST be derived server-side (random UUID + extension from sniffed MIME), never echoed from the client; a caller-supplied filename containing `../` or absolute paths is the canonical path-traversal sink for any local filesystem destination.
4. Uploaded user content served back to the browser MUST set `Content-Disposition: attachment` (or be hosted on a separate origin); serving user-uploaded HTML/SVG inline with `Content-Disposition: inline` from the same origin as your auth cookies is a one-step XSS.
5. Uploaded files MUST be scanned for known-bad signatures (ClamAV, Cloud-platform virus-scan, or equivalent) before they are made retrievable; a single malicious upload cached by your CDN means every visitor downloads the malware until the entry expires.
6. Uploaded files MUST NOT be stored under a path that includes the original filename, even sanitised; a deterministic content-addressed path (`<sha256>/file`) lets you dedupe uploads and prevents an attacker from probing for known names.
