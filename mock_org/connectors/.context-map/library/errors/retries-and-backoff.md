# Constraints — errors (retries and backoff)

1. Retry loops MUST use exponential backoff with full jitter (`sleep = random_between(0, base * 2 ** attempt)`); fixed-interval retries cause synchronized retry storms that knock dependencies over.
2. Every retry loop MUST have a max-attempts ceiling (typically 3–6) AND a max-elapsed wall-clock ceiling (typically 30–60 s); without both, a slow upstream that returns 503 forever pins the worker until the process is killed.
3. Idempotent operations MUST NOT be retried on 4xx responses other than 408 (timeout) and 429 (rate-limit); 400/401/403/404/422 are caller errors and retrying them only burns budget.
4. `catch` blocks MUST NOT silently swallow errors; either re-throw, log with the structured logger, or send to a dead-letter sink. A bare `catch {}` that does nothing is a known production-incident sink.
5. Every outbound HTTP call MUST set an explicit timeout (typically 5–15 s for sync, 30–120 s for streaming); a missing timeout inherits the OS default (often >2 minutes), and a single slow upstream is enough to exhaust your event loop.
6. Background-job failures that exceed the retry ceiling MUST land in a dead-letter sink (queue, table, or alert); jobs that silently disappear on final retry are the most common form of `we lost some user data and didn't notice for a week`.
