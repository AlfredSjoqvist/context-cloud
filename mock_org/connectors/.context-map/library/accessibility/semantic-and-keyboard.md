# Constraints — accessibility (semantic + keyboard)

1. Interactive elements MUST use semantic HTML (`<button>`, `<a href>`, `<input>`) before reaching for `<div role="button">`; the native element gives keyboard focus, Enter/Space activation, and screen-reader announcement for free, and a div with role+onClick gives you none of them by default.
2. Every interactive control MUST be reachable and activatable with the keyboard alone (Tab to focus, Enter or Space to activate, Esc to dismiss modals); a control that requires a pointer is invisible to screen-reader and switch-control users.
3. ARIA labels MUST be added only when the visible text or semantic element does not already convey the control's purpose; redundant `aria-label` on a `<button>Delete</button>` is announced twice and competes with the visible label.
4. Modal dialogs MUST trap focus inside the dialog while open, restore focus to the opener on close, and respond to Esc; without focus management a keyboard user can tab into the page behind the modal and lose context.
5. Text against its background MUST meet WCAG AA contrast (4.5:1 for body text, 3:1 for large text and UI elements); below this, low-vision users lose the content entirely and reviewer tools (axe, Lighthouse) flag every page.
6. Images that convey information MUST have a non-empty `alt` describing that information; decorative images MUST set `alt=""` (not omit it) so screen readers know to skip rather than read out the file path.
7. Animations longer than 5 seconds or that auto-loop MUST honor `prefers-reduced-motion` (CSS media query or JS check); ignoring it triggers vestibular reactions for affected users and is the only WCAG criterion users feel as nausea.
