# Constraints — concurrency (locks, races, and ordering)

1. Distributed locks MUST use a primitive with a TTL (Redis SET NX EX, Postgres advisory lock with timeout, DynamoDB conditional write); a lock without a TTL becomes permanent the first time the holder crashes between acquire and release.
2. Lock release MUST run inside a `finally` (or equivalent), not at the end of the happy path; an exception between acquire and release leaks the lock for the duration of the TTL — and an attacker who can trigger the exception holds it indefinitely.
3. Async work MUST NOT be issued in a serial `for ... await` loop when the items are independent; serialised awaits multiply latency by N. Use `Promise.all` (Node), `asyncio.gather` (Python), or `Task.WhenAll` (.NET) — but cap concurrency with a semaphore so the dependency doesn't get DOSed by your fan-out.
4. Per-key ordering guarantees MUST be enforced explicitly (single-partition queue, in-process per-key serializer, FIFO with deduplication). `Promise.all` and `asyncio.gather` complete in arbitrary order, so any handler that assumed write-then-read ordering on the same key will see stale reads.
5. Cache writes MUST NOT happen inside a database transaction; if the transaction rolls back, the cache holds a value that doesn't exist in the database — and the next reader gets a phantom that survives until the cache TTL expires.
6. Job handlers MUST be safe to run twice with the same input (same outputs, no double side-effects); every queue eventually delivers a duplicate due to ack-loss windows, and a non-idempotent handler will silently double-apply on every transient failure.
