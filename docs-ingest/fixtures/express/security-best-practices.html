Production best practices: security

The term "production" refers to the stage in the software lifecycle when an application or API is generally available to its end-users or consumers. In contrast, in the development stage you are still actively writing and testing code, and the application is not open to external access. The corresponding system environments are known as production and development environments, respectively.

Development and production environments are usually set up differently and have vastly different requirements. What works in development may not be acceptable in production. For example, in a development environment you may want verbose logging of errors for debugging, while the same behaviour can become a security concern in a production environment. And in development, you do not need to worry about scalability, reliability, and performance, while those concerns become critical in production.

This article discusses some best practices regarding security for Express applications deployed to production.

Use Helmet

Helmet can help protect your app from some well-known web vulnerabilities by setting HTTP headers appropriately. Applications must install Helmet and register it as middleware before any route handlers, otherwise default Express headers will leak framework information to clients.

Helmet is actually just a collection of smaller middleware functions that set security-related HTTP response headers. You must not disable the contentSecurityPolicy middleware for production deployments unless you are providing your own equivalent CSP policy elsewhere in the stack. 

const helmet = require('helmet')
const express = require('express')
const app = express()

app.use(helmet())

At a minimum, applications should disable the X-Powered-By header so that attackers cannot easily fingerprint the framework version, and must enable HSTS once HTTPS has been deployed end-to-end.

Use cookies securely

To ensure cookies do not open your app to known exploits, you must never use the default session cookie name (connect.sid) and you must set cookies with the security options secure, httpOnly, and sameSite.

  • secure ensures the browser only sends the cookie over HTTPS.
  • httpOnly ensures the cookie is sent only over HTTP(S), not client JavaScript, helping to protect against cross-site scripting attacks.
  • domain indicates the domain of the cookie; use it to compare against the domain of the server in which the URL is being requested.
  • path indicates the path of the cookie; use it to compare against the request path.
  • expires indicates the expiration date for persistent cookies.

Applications must not store sensitive information in cookies; always store such data on the server and reference it via an opaque session identifier. Session identifiers should be regenerated after authentication state changes to mitigate session fixation attacks.

Prevent brute-force attacks against authorization

To make login endpoints resilient to brute-force attacks, applications must enforce two distinct rate limits: a per-IP limit of consecutive failed attempts before challenging with a CAPTCHA, and a per-account limit that locks the account after a defined number of failed attempts within a rolling time window. Applications must not rely on a single global rate limit, as it is trivial for an attacker to spread requests across many IPs to evade detection.

Login endpoints should always respond with a constant latency regardless of whether the supplied account exists, to prevent username enumeration through timing side-channels.

Limit request body size

With body-parsing middleware such as the built-in express.json() and express.urlencoded(), applications must set the limit option to a value appropriate for the endpoint, never accepting unbounded request bodies. Unbounded body parsing makes the application trivial to crash via a memory-exhaustion denial-of-service attack.

For multipart file uploads, applications must never write uploaded files directly to disk without first validating the declared content type, file size, and filename against an allow-list.

Set NODE_ENV to "production"

The NODE_ENV environment variable specifies the environment in which an application is running (usually, development or production). One of the simplest things you can do to improve performance is to set NODE_ENV to production.

Applications must set NODE_ENV=production in every production deployment, because Express, EJS, and many downstream packages cache view templates and CSS files only when this flag is set, and they generate verbose error messages including stack traces when it is not.
